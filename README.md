<a id="en-overview"></a>

### Overview

`react-prototype-system` is an Agent Skill for turning design evidence into a responsive, interactive, and verifiable React prototype system. It is not a React component library and does not impose a runtime framework on an existing application. Instead, it gives an AI coding agent a structured engineering workflow, reusable templates, deterministic auditors, and completion gates.

The skill defaults new or unconstrained implementations to TypeScript and Tailwind CSS. In an established repository, local framework, routing, styling, token, component, and testing conventions take precedence.

The main objective is to preserve a direct path from design intent to implementation:

```text
experience
  -> interaction flow
  -> states
  -> component contracts
  -> design tokens
  -> implementation
  -> browser evidence
```

Dependencies are built from lower layers to higher layers:

```text
tokens -> primitives -> patterns -> layouts -> screens -> flows
```

<a id="en-use-cases"></a>

### What the skill is for

Use the skill when you need to:

- reconstruct a React component, screen, or multi-screen flow from Figma
- turn screenshots or visual specifications into a responsive prototype
- preserve meaningful Figma component, variant, property, and token names
- map a Figma design system into Tailwind-compatible semantic tokens
- extend an existing prototype without collapsing its component architecture
- review an existing React implementation for visual fidelity and missing states
- audit asset formats, malformed SVG files, unsafe SVG content, and stale asset references
- review state ownership, mock scenarios, recovery behavior, accessibility, and responsiveness
- inspect async waterfalls, bundle boundaries, server/client data transfer, repeated requests, hydration, and costly re-renders
- produce browser-backed verification evidence instead of claiming fidelity from source code alone

<a id="en-principles"></a>

### Core principles

- Inspect the scoped design source and repository before changing code.
- Preserve coherent local conventions instead of forcing a parallel architecture.
- Preserve meaningful Figma names and infer only missing or autogenerated names.
- Build the design-token foundation before product components when a source design system exists.
- Keep reusable display components controlled by data, status, and callbacks.
- Keep mock data, navigation, side effects, and orchestration outside display components.
- Place every reusable export in a matching named file.
- Let the component own responsive behavior when its responsibility remains stable across breakpoints.
- Implement the primary path and at least one relevant alternate or recovery state.
- Use supplied design assets and identify persisted formats from their bytes or parseable SVG structure.
- Review performance in impact order, beginning with async waterfalls and initial bundle cost.
- Never claim fidelity, token coverage, responsiveness, accessibility, or performance without corresponding evidence.

<a id="en-modes"></a>

### Operating modes

| Mode | Use it when | Expected outcome |
| --- | --- | --- |
| `build` | Creating a component, screen, or flow from a design source | Implement and verify the scoped experience |
| `review` | Inspecting an existing implementation | Report evidence-backed findings; modify only when fixes are requested |
| `evolve` | Adding states, screens, or behavior to an existing prototype | Preserve working contracts while implementing and verifying the change |

The selected mode changes the amount of planning required. A substantial build may need a flow map, naming map, component inventory, token map, state matrix, asset map, and critical-path map. A focused review should collect only the evidence needed to support its findings.

<a id="en-evidence"></a>

### Evidence levels

| Level | Available evidence | Claim boundary |
| --- | --- | --- |
| `complete` | Scoped Figma nodes, names, variables or styles, assets, and screenshots | Source-traceable naming, tokens, assets, and visual comparison |
| `partial` | Frames are visible but variables, bindings, variants, or assets are incomplete | Claim only the inspected categories and list every gap |
| `screenshot-only` | Rasterized visual evidence without authoritative metadata | Claim visual interpretation, not Figma token or naming coverage |
| `unavailable` | The named design source cannot be inspected | Use repository evidence only and mark design verification `NOT RUN` |

Figma access and browser access are separate evidence surfaces. Complete Figma metadata does not prove that the rendered page matches the design, and a successful browser run does not recover unavailable Figma token bindings.

<a id="en-workflow"></a>

### Workflow

1. **Classify the task.** Select `build`, `review`, or `evolve`; record code context, design scope, fidelity target, and evidence level.
2. **Inspect and model.** Read the relevant repository conventions and only the scoped design nodes, assets, states, and breakpoints.
3. **Establish the visual foundation.** Reuse repository tokens; otherwise map source semantics into the supported Tailwind or CSS-variable mechanism.
4. **Build or change bottom-up.** Reuse primitives, create patterns and typed domain components, compose layouts and screens, then connect routes and state.
5. **Make states addressable.** Use deterministic fixtures and a query parameter, fixture route, story, test selector, or documented action sequence.
6. **Complete the interaction flow.** Connect navigation, dialogs, filters, forms, retry, undo, dismissal, and recovery behavior.
7. **Protect the critical path.** Remove avoidable waterfalls, defer heavy off-path modules, keep server/client boundaries narrow, and optimize renders only when evidence supports it.
8. **Verify and report.** Run the application at source, desktop, narrow, and stress viewports and record `PASS`, `FAIL`, `NOT RUN`, or `NOT APPLICABLE`.

<a id="en-tools"></a>

### Bundled tools

#### Asset auditor

Detect image formats from content, validate raster structure, inspect SVG safety and geometry, and report suffix mismatches:

```bash
python3 react-prototype-system/scripts/audit_figma_assets.py src/assets
```

Review the report before applying safe filename repairs:

```bash
python3 react-prototype-system/scripts/audit_figma_assets.py \
  --fix \
  --references-root . \
  src/assets
```

Recheck source references after a previous rename:

```bash
python3 react-prototype-system/scripts/audit_figma_assets.py \
  --references-root . \
  --reference-change src/assets/SearchIcon.png=src/assets/SearchIcon.svg \
  src/assets/SearchIcon.svg
```

The fixer renames files only. It never overwrites an existing target, sanitizes SVG markup, or rewrites code imports automatically.

#### Design-token auditor

Copy [`design-token-map.example.json`](react-prototype-system/assets/design-token-map.example.json), populate it with scoped source evidence and exact consumers, then run:

```bash
python3 react-prototype-system/scripts/audit_design_tokens.py \
  path/to/design-token-map.json \
  --root path/to/project
```

The auditor validates schema, source access, modes, duplicate identities, consumer files, declared usages, one-off exceptions, common raw Tailwind palette utilities, visual arbitrary utilities, and CSS color literals.

#### Component scaffolder

Use only in a greenfield project without an existing component generator:

```bash
python3 react-prototype-system/scripts/create_component.py ProductCard \
  --dir src/components \
  --element article \
  --index
```

The scaffolder creates a small stateless TypeScript component, matching filename and export, typed host-element props, and an optional `index.ts` without overwriting existing files.

<a id="en-verification"></a>

### Verification

Run the bundled tests:

```bash
python3 -m unittest discover \
  -s react-prototype-system/scripts/tests \
  -p 'test_*.py'
```

For prototype work, copy [`verification-report.md`](react-prototype-system/assets/verification-report.md) into a temporary project note and record evidence with these statuses:

- `PASS`: directly observed or supported by command output
- `FAIL`: executed and unsuccessful
- `NOT RUN`: required evidence or tooling was unavailable
- `NOT APPLICABLE`: outside the scoped mode or repository

Visual fidelity must be checked in a running browser. A passing build or source-code review is not proof of responsive correctness, asset rendering, focus behavior, or design fidelity.

<a id="en-requirements"></a>

### Requirements and limits

- The skill requires an agent environment that supports Agent Skills.
- The bundled auditors use the Python standard library and require a modern Python 3 installation.
- React, TypeScript, Tailwind, browser, and testing dependencies belong to the target project; the skill does not install a parallel application stack.
- The skill does not provide Figma credentials or an independent Figma API connection. It uses the design access available to the agent.
- The token auditor proves manifest consistency and declared consumer evidence; it cannot prove that inaccessible Figma bindings were inventoried.
- Raw-value scanning covers common Tailwind and CSS patterns but is not a complete CSS parser.
- Browser verification remains necessary even when asset and token audits pass.
- Review mode does not authorize code changes unless the request also asks for fixes.

<a id="zh-overview"></a>

### 概述

`react-prototype-system` 是一套 Agent Skill，用来将设计证据转换为响应式、可交互并且可以验证的 React prototype system。它不是 React 组件库，也不会强制已有应用更换运行时框架。它为 AI coding agent 提供结构化工程流程、可复用模板、确定性审计工具和明确的完成门槛。

对于新项目或未指定实现方式的任务，skill 默认使用 TypeScript 和 Tailwind CSS。对于已有项目，现有的框架、路由、样式、Token、组件和测试规范优先。

它希望在设计意图与实现之间保持清晰的追溯链路：

```text
体验
  -> 交互流程
  -> 状态
  -> 组件契约
  -> 设计 Token
  -> 实现
  -> 浏览器证据
```

依赖关系按照从底层到上层的顺序构建：

```text
tokens -> primitives -> patterns -> layouts -> screens -> flows
```

<a id="zh-use-cases"></a>

### 适用场景

以下任务适合使用该 skill：

- 根据 Figma 还原 React 组件、页面或多页面流程
- 将截图或视觉规范转换为响应式 prototype
- 保留有意义的 Figma 组件、Variant、Property 和 Token 命名
- 将 Figma Design System 映射为 Tailwind 兼容的语义 Token
- 在不破坏组件架构的情况下扩展现有 prototype
- 检查已有 React 实现的视觉还原度和缺失状态
- 审计资源真实格式、损坏或不安全的 SVG，以及资源重命名后的遗留引用
- 检查状态归属、Mock 场景、恢复行为、无障碍与响应式
- 检查异步瀑布、Bundle 边界、Server/Client 数据传输、重复请求、Hydration 和高成本重渲染
- 使用浏览器证据完成验证，而不是只根据源代码声称高保真

<a id="zh-principles"></a>

### 核心原则

- 修改代码之前先检查任务范围内的设计来源和代码仓库。
- 保留清晰一致的本地规范，不引入平行架构。
- 保留有意义的 Figma 命名，只为缺失或自动生成的名称进行语义推断。
- 如果设计来源包含 Design System，先建设 Token 基础，再实现产品组件。
- 可复用展示组件通过明确的数据、状态和 Callback 控制。
- Mock、导航、副作用和流程编排放在展示组件之外。
- 每个可复用导出组件使用同名文件。
- 当组件职责在不同断点保持一致时，由组件自身负责响应式行为。
- 实现主要流程以及至少一个相关的备选、失败或恢复状态。
- 使用设计提供的真实资源，并根据文件字节或可解析 SVG 结构确定持久化格式。
- 按影响顺序检查性能，先处理异步瀑布和初始 Bundle。
- 没有对应证据时，不声称完成了视觉还原、Token 覆盖、响应式、无障碍或性能优化。

<a id="zh-modes"></a>

### 工作模式

| 模式 | 适用情况 | 预期结果 |
| --- | --- | --- |
| `build` | 根据设计来源创建组件、页面或流程 | 实现并验证任务范围内的体验 |
| `review` | 检查已有实现 | 输出有证据支持的问题；只有用户要求修复时才修改代码 |
| `evolve` | 为已有 prototype 增加状态、页面或行为 | 保留已有契约，完成变更并验证 |

不同模式需要不同深度的规划。较大的 build 任务可能需要流程图、命名映射、组件清单、Token 映射、状态矩阵、资源表和关键路径图；聚焦的 review 只需要收集能够证明问题的证据。

<a id="zh-evidence"></a>

### 证据等级

| 等级 | 可用证据 | 可声明范围 |
| --- | --- | --- |
| `complete` | 任务范围内的 Figma 节点、命名、Variables 或 Styles、资源和截图 | 可追溯的命名、Token、资源和视觉对比 |
| `partial` | 可以查看画面，但 Variables、Bindings、Variants 或资源不完整 | 只声明已经检查的类别，并列出所有缺口 |
| `screenshot-only` | 只有位图视觉证据，没有权威元数据 | 只能声明视觉解释，不能声明 Figma Token 或命名覆盖 |
| `unavailable` | 无法检查指定设计来源 | 只能使用仓库证据，并将设计验证记为 `NOT RUN` |

Figma 访问与浏览器访问是两种独立证据。完整的 Figma 元数据不能证明实际页面已经正确渲染；浏览器可以正常运行，也不能补回无法访问的 Figma Token Binding。

<a id="zh-workflow"></a>

### 工作流程

1. **分类任务。** 选择 `build`、`review` 或 `evolve`，记录代码环境、设计范围、还原目标和证据等级。
2. **检查并建模。** 阅读相关仓库规范，只检查任务范围内的设计节点、资源、状态和断点。
3. **建立视觉基础。** 优先复用仓库 Token，否则将设计语义映射到项目支持的 Tailwind 或 CSS Variable 机制。
4. **自底向上构建或修改。** 复用 Primitive，构建 Pattern 和带类型的领域组件，组合 Layout 和 Screen，最后连接路由和状态。
5. **让状态可寻址。** 使用确定性 Fixture，并通过 Query Parameter、Fixture Route、Story、测试选择器或清晰操作步骤访问场景。
6. **完成交互流程。** 连接导航、Dialog、筛选、表单、重试、撤销、关闭和恢复行为。
7. **保护关键路径。** 消除可避免的瀑布，延迟首屏外重型模块，缩小 Server/Client 边界，只针对有证据的成本优化渲染。
8. **验证并记录。** 在设计尺寸、桌面、窄屏和压力场景下运行应用，并记录 `PASS`、`FAIL`、`NOT RUN` 或 `NOT APPLICABLE`。

### 内置工具

#### 资源审计器

根据内容识别图片格式、检查光栅文件结构、审计 SVG 安全性和几何结构，并报告后缀不匹配：

```bash
python3 react-prototype-system/scripts/audit_figma_assets.py src/assets
```

先阅读报告，再进行安全的文件名修复：

```bash
python3 react-prototype-system/scripts/audit_figma_assets.py \
  --fix \
  --references-root . \
  src/assets
```

重新检查历史重命名对应的代码引用：

```bash
python3 react-prototype-system/scripts/audit_figma_assets.py \
  --references-root . \
  --reference-change src/assets/SearchIcon.png=src/assets/SearchIcon.svg \
  src/assets/SearchIcon.svg
```

修复器只修改文件名，不会覆盖已有目标、清理 SVG 内容或自动改写代码 Import。

#### Design Token 审计器

复制 [`design-token-map.example.json`](react-prototype-system/assets/design-token-map.example.json)，填入任务范围内的来源证据和精确消费者，然后运行：

```bash
python3 react-prototype-system/scripts/audit_design_tokens.py \
  path/to/design-token-map.json \
  --root path/to/project
```

审计器会检查 Schema、来源访问等级、Modes、重复身份、消费者文件、声明的 Usage、一次性例外、常见 Tailwind 原始色板、视觉 Arbitrary Utility 和 CSS 颜色字面量。

#### 组件脚手架

只在没有现成组件生成器的新项目中使用：

```bash
python3 react-prototype-system/scripts/create_component.py ProductCard \
  --dir src/components \
  --element article \
  --index
```

脚手架会创建最小化无状态 TypeScript 组件、匹配的文件名与导出、带类型的宿主元素 Props，以及可选的 `index.ts`，并且拒绝覆盖已有文件。

<a id="zh-requirements"></a>

### 要求与限制

- 需要支持 Agent Skills 的 agent 环境。
- 内置审计器只使用 Python 标准库，需要现代 Python 3 环境。
- React、TypeScript、Tailwind、浏览器和测试依赖属于目标项目；skill 不会安装平行应用栈。
- skill 不提供 Figma 凭据或独立 Figma API 连接，只使用 agent 当前可用的设计访问能力。
- Token 审计器可以证明 Manifest 内部一致性和已声明的消费者证据，但不能证明无法访问的 Figma Binding 已被完整盘点。
- Raw Value 扫描覆盖常见 Tailwind 和 CSS 模式，但不是完整 CSS Parser。
- 即使资源和 Token 审计通过，仍然需要浏览器验证。
- `review` 模式不会自动授权修改代码，除非请求同时要求修复。

<a id="ja-overview"></a>

### 概要

`react-prototype-system` は、デザイン根拠をレスポンシブで操作可能、かつ検証可能な React prototype system に変換するための Agent Skill です。React コンポーネントライブラリではなく、既存アプリケーションに特定のランタイムフレームワークを強制するものでもありません。AI coding agent に対して、構造化されたエンジニアリング手順、再利用可能なテンプレート、決定的な監査ツール、明確な完了条件を提供します。

新規または実装方法が未指定のタスクでは、TypeScript と Tailwind CSS を標準とします。既存リポジトリでは、フレームワーク、ルーティング、スタイル、Token、コンポーネント、テストのローカル規約を優先します。

デザイン意図から実装まで、次の追跡可能な経路を維持します。

```text
体験
  -> インタラクションフロー
  -> 状態
  -> コンポーネント契約
  -> デザイン Token
  -> 実装
  -> ブラウザエビデンス
```

依存関係は下位レイヤーから上位レイヤーの順に構築します。

```text
tokens -> primitives -> patterns -> layouts -> screens -> flows
```

<a id="ja-use-cases"></a>

### 利用場面

次のようなタスクで使用します。

- Figma から React コンポーネント、画面、複数画面フローを再構築する
- スクリーンショットやビジュアル仕様をレスポンシブな prototype に変換する
- 意味のある Figma コンポーネント、Variant、Property、Token 名を維持する
- Figma Design System を Tailwind 互換のセマンティック Token に変換する
- コンポーネントアーキテクチャを崩さず既存 prototype を拡張する
- 既存 React 実装のデザイン忠実度や不足状態をレビューする
- アセットの実形式、破損または危険な SVG、リネーム後の古い参照を監査する
- 状態の所有範囲、Mock シナリオ、復旧動作、アクセシビリティ、レスポンシブ対応を確認する
- 非同期ウォーターフォール、Bundle 境界、Server/Client 間の転送、重複リクエスト、Hydration、高コストな再レンダリングを確認する
- ソースコードだけで忠実度を宣言せず、ブラウザに基づく検証エビデンスを残す

<a id="ja-principles"></a>

### 基本原則

- コード変更前に、対象範囲のデザインソースとリポジトリを確認します。
- 一貫したローカル規約を維持し、並行する別アーキテクチャを導入しません。
- 意味のある Figma 名を維持し、欠落名や自動生成名のみを文脈から推定します。
- ソースに Design System がある場合は、製品コンポーネントより先に Token 基盤を構築します。
- 再利用可能な表示コンポーネントは、明示的なデータ、状態、Callback で制御します。
- Mock、ナビゲーション、副作用、フロー制御を表示コンポーネントの外に置きます。
- 再利用可能な Export は、同名ファイルに配置します。
- ブレークポイント間で責務が同じなら、レスポンシブ動作はコンポーネント自身が持ちます。
- メインフローと、関連する代替、失敗、または復旧状態を少なくとも 1 つ実装します。
- デザインから提供されたアセットを使用し、バイト列または解析可能な SVG 構造から保存形式を判定します。
- パフォーマンスは影響順に確認し、非同期ウォーターフォールと初期 Bundle から扱います。
- 対応するエビデンスなしに、忠実度、Token カバレッジ、レスポンシブ、アクセシビリティ、性能改善を宣言しません。

<a id="ja-modes"></a>

### 動作モード

| モード | 使用場面 | 期待する結果 |
| --- | --- | --- |
| `build` | デザインソースからコンポーネント、画面、フローを作成する | 対象体験を実装して検証する |
| `review` | 既存実装を調査する | エビデンス付きの指摘を報告し、修正依頼がある場合のみ変更する |
| `evolve` | 既存 prototype に状態、画面、動作を追加する | 既存契約を維持しながら変更を実装して検証する |

選択したモードによって必要な計画量が変わります。大規模な build では、フローマップ、命名マップ、コンポーネント一覧、Token マップ、状態マトリクス、アセット表、クリティカルパスマップが必要になる場合があります。限定的な review では、指摘を証明するために必要なエビデンスだけを収集します。

<a id="ja-evidence"></a>

### エビデンスレベル

| レベル | 利用可能なエビデンス | 宣言可能な範囲 |
| --- | --- | --- |
| `complete` | 対象 Figma ノード、命名、Variables または Styles、アセット、スクリーンショット | ソース追跡可能な命名、Token、アセット、視覚比較 |
| `partial` | 画面は確認できるが、Variables、Bindings、Variants、アセットが不完全 | 確認済みカテゴリだけを宣言し、欠落をすべて記録する |
| `screenshot-only` | 権威あるメタデータがないラスター画像 | 視覚的解釈のみを宣言し、Figma Token や命名カバレッジは宣言しない |
| `unavailable` | 指定デザインソースを確認できない | リポジトリエビデンスだけを使用し、デザイン検証を `NOT RUN` とする |

Figma へのアクセスとブラウザへのアクセスは別々のエビデンスです。完全な Figma メタデータだけではレンダリング結果を証明できず、ブラウザが正常に動いてもアクセスできない Figma Token Binding は復元できません。

<a id="ja-workflow"></a>

### ワークフロー

1. **タスクを分類する。** `build`、`review`、`evolve` を選び、コード環境、デザイン範囲、忠実度目標、エビデンスレベルを記録します。
2. **調査してモデル化する。** 関連するリポジトリ規約を読み、対象範囲のデザインノード、アセット、状態、ブレークポイントだけを確認します。
3. **ビジュアル基盤を作る。** 既存 Token を優先し、必要であればソースの意味をプロジェクトの Tailwind または CSS Variable 方式へマッピングします。
4. **下位層から構築または変更する。** Primitive を再利用し、Pattern と型付きドメインコンポーネントを作り、Layout と Screen を構成してからルートと状態を接続します。
5. **状態を参照可能にする。** 決定的な Fixture と、Query Parameter、Fixture Route、Story、テストセレクター、または明確な操作手順を使用します。
6. **インタラクションフローを完成させる。** ナビゲーション、Dialog、フィルター、フォーム、再試行、Undo、Dismiss、復旧動作を接続します。
7. **クリティカルパスを守る。** 不要なウォーターフォールを除去し、初期表示外の重いモジュールを遅延し、Server/Client 境界を狭くし、根拠のあるコストだけを最適化します。
8. **検証して記録する。** ソース寸法、デスクトップ、狭い画面、ストレスシナリオでアプリを実行し、`PASS`、`FAIL`、`NOT RUN`、`NOT APPLICABLE` を記録します。

### 使用例

デザインから構築する：

```text
$react-prototype-system の build モードを使用して、この Figma デザインを
```

<a id="ja-tools"></a>

### 同梱ツール

#### アセット監査ツール

内容から画像形式を判定し、ラスター構造、SVG の安全性と形状、拡張子不一致を確認します。

```bash
python3 react-prototype-system/scripts/audit_figma_assets.py src/assets
```

レポートを確認してから、安全なファイル名修正を実行します。

```bash
python3 react-prototype-system/scripts/audit_figma_assets.py \
  --fix \
  --references-root . \
  src/assets
```

以前のリネームに対するコード参照を再確認します。

```bash
python3 react-prototype-system/scripts/audit_figma_assets.py \
  --references-root . \
  --reference-change src/assets/SearchIcon.png=src/assets/SearchIcon.svg \
  src/assets/SearchIcon.svg
```

修正機能はファイル名だけを変更します。既存ターゲットの上書き、SVG のサニタイズ、コード Import の自動書き換えは行いません。

#### Design Token 監査ツール

[`design-token-map.example.json`](react-prototype-system/assets/design-token-map.example.json) をコピーし、対象範囲のソース根拠と正確な利用箇所を記入して実行します。

```bash
python3 react-prototype-system/scripts/audit_design_tokens.py \
  path/to/design-token-map.json \
  --root path/to/project
```

Schema、ソースアクセスレベル、Modes、重複 ID、利用ファイル、宣言 Usage、単発例外、一般的な Tailwind の raw palette、視覚 arbitrary utility、CSS 色リテラルを確認します。

#### コンポーネントスキャフォルダー

既存ジェネレーターがない greenfield プロジェクトでのみ使用します。

```bash
python3 react-prototype-system/scripts/create_component.py ProductCard \
  --dir src/components \
  --element article \
  --index
```

小さなステートレス TypeScript コンポーネント、Export と一致するファイル名、型付きホスト要素 Props、任意の `index.ts` を作成し、既存ファイルは上書きしません。

<a id="ja-verification"></a>

### 検証

同梱テストを実行します。

```bash
python3 -m unittest discover \
  -s react-prototype-system/scripts/tests \
  -p 'test_*.py'
```

prototype 作業では、[`verification-report.md`](react-prototype-system/assets/verification-report.md) を一時的なプロジェクトノートへコピーし、次の状態でエビデンスを記録します。

- `PASS`：直接確認済み、またはコマンド出力で裏付けられている
- `FAIL`：実行したが失敗した
- `NOT RUN`：必要なエビデンスまたはツールが利用できなかった
- `NOT APPLICABLE`：現在のモードまたはリポジトリ範囲外

視覚的忠実度は実行中のブラウザで確認する必要があります。Build 成功やソースレビューだけでは、レスポンシブの正しさ、アセット表示、Focus 動作、デザイン忠実度を証明できません。

<a id="ja-requirements"></a>

### 要件と制限

- Agent Skills をサポートする agent 環境が必要です。
- 同梱監査ツールは Python 標準ライブラリだけを使用し、モダンな Python 3 が必要です。
- React、TypeScript、Tailwind、ブラウザ、テスト依存関係は対象プロジェクト側に属し、skill は別のアプリケーションスタックを導入しません。
- Figma 認証情報や独立した Figma API 接続は含まれません。agent が利用可能なデザインアクセスを使用します。
- Token 監査は Manifest の整合性と宣言済み利用箇所を証明しますが、アクセス不能な Figma Binding の完全な棚卸しは証明できません。
- Raw value スキャンは一般的な Tailwind と CSS パターンを対象としますが、完全な CSS Parser ではありません。
- アセット監査と Token 監査が成功しても、ブラウザ検証は必要です。
- `review` モードは、修正依頼が同時にない限りコード変更を許可しません。
