# React Prototype System

Build the experience as a system, then compose the screens. Optimize for learning, interaction, fidelity, and changeability while leaving a clear path to production.

## Core Contract

Apply these rules in every mode:

1. Inspect the scoped design evidence and repository before changing code.
2. Preserve coherent local framework, routing, styling, token, component, and test conventions.
3. Default unconstrained implementations to TypeScript and Tailwind CSS; do not migrate an established stack without a request.
4. Preserve meaningful Figma names after code-safe normalization. Infer only missing or autogenerated names from product role and context.
5. Map a detected Figma design system into auditable semantic tokens before product components consume it. Use Tailwind defaults when no design system exists.
6. Build in dependency order: `tokens -> primitives -> patterns -> layouts -> screens -> flows`.
7. Put each reusable export in a matching named file and make the component, rather than the page, own responsive behavior.
8. Keep reusable display components controlled by explicit data, status, and callbacks. Keep mocks, navigation, side effects, and orchestration outside them.
9. Use supplied assets, verify persisted formats from content, and never substitute inspectable product content with vague placeholders.
10. Make the primary flow and one relevant alternate or recovery state reachable and deterministic.
11. Review performance in impact order: waterfalls, initial bundle, server/client boundaries, duplicate client work, re-renders, browser rendering, then JavaScript hot paths.
12. Claim only what the available design and browser evidence proves.

Treat these as defaults, not permission to fight a coherent codebase. Stop adding architecture when it no longer improves fidelity, responsiveness, reuse, or changeability.

## Choose an Operating Mode

Select one primary mode before gathering evidence:

| Mode | Use when | Required outcome |
| --- | --- | --- |
| `build` | Creating a component, screen, or flow from a design source | Implement and verify the scoped experience |
| `review` | Inspecting an existing implementation | Report evidence-backed findings; modify only when the user also requests fixes |
| `evolve` | Extending an existing prototype or adding states, screens, or behaviors | Preserve working contracts while implementing and verifying the change |

Also record the code context, design scope, and fidelity target. If fidelity is unspecified, reproduce the supplied design closely.

Mode affects process depth:

- In `build`, create the minimum useful maps before JSX and implement bottom-up.
- In `review`, inspect only the evidence needed to prove findings; do not generate planning artifacts by habit.
- In `evolve`, map the affected contracts and dependencies, then keep changes local to the smallest responsible layers.

## Establish the Evidence Level

Record one design-evidence level and constrain claims accordingly:

| Level | Available evidence | Allowed claims |
| --- | --- | --- |
| `complete` | Scoped Figma nodes, names, variables/styles, assets, and screenshots | Source-traceable naming, tokens, assets, and visual comparison |
| `partial` | Visual nodes plus incomplete variables, bindings, assets, or variants | Claim only the inspected categories; list gaps |
| `screenshot-only` | Rasterized visual evidence without authoritative Figma metadata | Claim visual interpretation, not source token or naming coverage |
| `unavailable` | The named design or browser evidence cannot be inspected | Diagnose repository evidence only; do not claim design fidelity |

If browser execution is unavailable, report rendered behavior, responsiveness, and visual fidelity as `NOT RUN`, not passed. Read [design-intake.md](references/design-intake.md) for source priority, access decisions, and inference rules.

## Workflow

### 1. Inspect and Model

Inspect the repository for routes, framework, TypeScript and styling setup, tokens, components, icons, assets, state/data conventions, and supported checks. Inspect only the design nodes, variants, breakpoints, and assets needed for the scoped experience.

For substantial `build` work, copy [prototype-brief.md](assets/prototype-brief.md) into a temporary project note and fill only useful sections. Model:

- screen or flow transitions
- Figma-to-code names and any public inferred names
- component layers and ownership
- design-system/token decisions
- required scenarios and recovery
- assets and responsive constraints
- first useful render and material performance risks

For `review` and narrow `evolve` tasks, record only the artifacts needed to support the requested outcome.

### 2. Establish the Visual Foundation

Reuse equivalent repository tokens first. If the scoped Figma source exposes a design system:

1. inventory scoped primitive values, semantic aliases, modes, styles, and component properties
2. copy [design-token-map.example.json](assets/design-token-map.example.json) to a project-local working manifest
3. map semantics into the Tailwind mechanism already supported by the repository
4. preserve primitive-to-semantic relationships and meaningful source names
5. run [audit_design_tokens.py](scripts/audit_design_tokens.py) after implementation

If no design system exists, use Tailwind defaults as the baseline and add only evidence-backed extensions. Do not create a manifest that presents inferred tokens as Figma-sourced.

Read [figma-naming-and-tokens.md](references/figma-naming-and-tokens.md) for naming and mapping decisions and [design-token-audit.md](references/design-token-audit.md) when a design-system manifest is required.

For every persisted design asset, follow [figma-assets.md](references/figma-assets.md). Before importing it, run [audit_figma_assets.py](scripts/audit_figma_assets.py) in read-only mode. Use `--fix` only after reviewing safe suffix changes. When renaming, pass `--references-root` and resolve every stale reference before completion.

### 3. Build or Change Bottom-Up

In `build` and `evolve` modes:

1. reuse or extend primitives
2. build reusable patterns and typed domain components
3. compose layouts and screens
4. connect routes, data ownership, callbacks, and state transitions

Normalize meaningful Figma component names to PascalCase and match the export and `.tsx` filename. Preserve variant axes as typed props. Infer semantic names only for absent or generic source names.

Keep screens focused on `import -> compose -> connect`. Make responsive behavior live in the named component unless its content or interaction contract materially changes. Read [architecture.md](references/architecture.md) for placement and [component-contracts.md](references/component-contracts.md) for public APIs and semantics.

Use [create_component.py](scripts/create_component.py) only for a greenfield component when the repository has no conflicting generator or file convention.

### 4. Make States and Flows Deliberate

Place state at the narrowest owner shared by its consumers. Derive values instead of mirroring them. Represent mutually exclusive statuses explicitly.

Keep prototype fixtures deterministic. Expose required scenarios through an addressable mechanism such as a query parameter, dedicated fixture route, or test harness when doing so will not alter the product UI. Include the primary populated state and one relevant failure, empty, permission, or recovery state.

Make every interactive-looking control work or expose an explicit disabled state. Connect navigation, overlays, filters, forms, retry, undo, and dismissal as required. Read [state-and-mocks.md](references/state-and-mocks.md) and [flows-motion-responsive.md](references/flows-motion-responsive.md).

### 5. Protect the Critical Path

After adding several components, real async work, a server/client boundary, or a heavy dependency, review the changed route as one runtime path:

1. start independent work together
2. defer genuinely heavy off-path modules
3. keep server/client transfer and request state narrow
4. reuse the existing client data layer or deterministic local mocks
5. remove duplicated derived state and action-through-effect patterns
6. add memoization, transitions, virtualization, or hot-path changes only for evidenced cost

Preserve stable loading geometry and existing framework capabilities. Read [react-performance.md](references/react-performance.md) for the detailed review policy.

### 6. Verify and Report Evidence

Run the intended entry point and exercise the complete scoped path. Compare the rendered interface with the source at the exact design viewport when known, one representative desktop width, one narrow width, and a relevant stress scenario.

For substantial work, copy [verification-report.md](assets/verification-report.md) into a temporary project note. Record every applicable check as:

- `PASS` — directly observed or supported by command output
- `FAIL` — executed and unsuccessful
- `NOT RUN` — required evidence or tooling was unavailable
- `NOT APPLICABLE` — outside the scoped mode or repository

Inspect visual hierarchy, assets, typography, states, focus and keyboard behavior, reduced motion, console and network output, async ordering, hydration, and supported tests/typecheck/lint/build. Use existing accessibility automation such as Playwright or axe when already installed; do not add a test stack solely for the prototype.

Read [verification.md](references/verification.md) before declaring completion.

## Mode Completion Gates

### Build and Evolve

- The intended route renders without runtime errors.
- The scoped primary path and one relevant alternate or recovery path work.
- Reusable components, state, data, and dependency boundaries follow the local architecture.
- Persisted assets pass the asset audit and have no stale renamed references.
- A required design-token manifest passes its audit, or the evidence record explains why no source design system applies.
- Desktop, narrow, and stress scenarios were inspected, or are explicitly `NOT RUN`.
- Supported engineering checks pass, or failures are reported with cause.

### Review

- Every finding cites a file, rendered state, command result, or reproducible interaction.
- Findings are ordered by broken flow, major fidelity/state gaps, architecture, responsive/accessibility regressions, then maintainability.
- Unverified assumptions and unavailable evidence are explicit.
- No fixes are made unless the user requested implementation.

## Guardrails

- Do not build a monolithic page when repeated or independently interactive regions have stable contracts.
- Do not fetch, navigate, or mutate unrelated global state inside reusable display components.
- Do not allow boolean combinations that represent impossible states.
- Do not trust export filenames over asset bytes or parseable SVG structure.
- Do not hardcode semantic visual values that bypass an available mapped design system.
- Do not duplicate desktop and mobile markup when one responsive component keeps the same contract.
- Do not add production-scale infrastructure, speculative abstractions, or unevidenced optimization to a prototype.
- Do not claim pixel fidelity, responsive correctness, token coverage, accessibility, or performance improvement without the corresponding evidence.
